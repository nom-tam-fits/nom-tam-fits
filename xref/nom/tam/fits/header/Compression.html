295     /**
296      * Gzip is the compression algorithm used in the free GN U software utility

298      * is based on the DEFLATE algorithm, which is a combination of LZ77 and
299      * Huffman coding. DEFLATE was intended as a replacement for LZW and other
300      * patent-encumbered data compression algor ithms which, at the time,
301      * limited the usability of compress and other popular archivers. Furt her
302      * information about this compression technique is readily available on the
303      * Internet. The gzip alg orithm has no associated parameters that need to
304      * be specified with the ZNAMEn and ZVALn keywords.
305      */
306     public static final String ZCMPTYPE_GZIP_1 = "GZIP_1";

308     /**
309      * If ZCMPTYPE = ’GZIP 2’ then the bytes in the array of image pixel values
310      * are shuffled in to decreasing order of significance before being
311      * compressed with the gzip algorithm. In other words, bytes are shuffled so
312      * that the most significant byte of every pixel occurs first, in order,
313      * followed by the next most significant byte, and so on for every byte.
314      * Since the most significan bytes of the pixel values often have very
315      * similar values, grouping them together in this way often achieves better
316      * net compression of the array. This is usually especially effective when
317      * compressing floating-point arrays.
318      */
319     public static final String ZCMPTYPE_GZIP_2 = "GZIP_2";
320 
321     /**
322      * If ZCMPTYPE = ’RICE 1’ then the Rice algorithm is used to compress and
323      * uncompress the image pixels. The Rice algorithm (Rice, R. F., Yeh, P.-S.,
324      * and Miller, W. H. 1993, in Proc. of the 9th AIAA Computing in Aerospace
325      * Conf., AIAA-93-4541-CP, American Institute of Aeronautics and
326      * Astronautics) is simple and very fast, compressing or decompressing 10 7
327      * pixels/sec on modern workstations. It requires only enough memory to hold
328      * a single block of 16 or 32 pixels at a time. It codes the pixels in small
329      * blocks and so is able to adapt very quickly to changes in the input image
330      * statistics (e.g., Rice has no problem handling cosmic rays, bright stars,
331      * saturated pixels, etc.).
332      */
333     public static final String ZCMPTYPE_RICE_1 = "RICE_1";
334 
335     /**
336      * If ZCMPTYPE = ’PLIO 1’ then the IRAF PLIO (Pixel List) algorithm is used
337      * to compress and uncompress the image pixels. The PLIO algorithm was
338      * developed to store integer-valued image masks in a compressed form.
339      * Typical uses of image masks are to segment images into regions, or to
340      * mark bad pixels. Such masks often have large regions of constant value
341      * hence are highly compressible. The compression algorithm used is based on
342      * run-length encoding, with the ability to dynamically follow level changes
343      * in the image, allowing a 16-bit encoding to be used regardless of the
344      * image depth. The worst case performance occurs when successive pixels
345      * have different values. Even in this case the encoding will only require
346      * one word (16 bits) per mask pixel, provided either the delta intensity
347      * change between pixels is usually less than 12 bits, or the mask
348      * represents a zero floored step function of constant height. The worst
349      * case cannot exceed npix*2 words provided the mask depth is 24 bits or
350      * less.
351      */
352     public static final String ZCMPTYPE_PLIO_1 = "PLIO_1";
353 
354     /**
355      * Hcompress is an the image compression package written by Richard L. White
356      * for use at the Space Telescope Science Institute. Hcompress was used to
357      * compress the STScI Digitized Sky Survey and has also been used to
358      * compress the preview images in the Hubble Data Archive. Briefly, the
359      * method used is: <br>
360      * 1. a wavelet transform called the H-transform (a Haar transform
361      * generalized to two dimensions), followed by<br>
362      * 2. quantization that discards noise in the image while retaining the
363      * signal on all scales, followed by 10<br>
364      * 3. quadtree coding of the quantized coefficients.<br>
365      * The technique gives very good compression for astronomical images and is
366      * relatively fast. The calculations are carried out using integer
367      * arithmetic and a re entirely reversible. Consequently, the program can be
368      * used for either lossy or lossless compression , with no special approach
369      * needed for the lossless case (e.g. there is no need for a file of
370      * residuals .)
371      */
372     public static final String ZCMPTYPE_HCOMPRESS_1 = "HCOMPRESS_1";

439     /**
440      * <p>
441      * The null pixels in integer images are flagged by a reserved BLANK value
442      * and will be preserved if a lossless compression algorithm is used. If the
443      * image is compressed with a lossy algorithm, however (e.g., H-Compress
444      * with a scale factor greater than 1), then some other technique must be
445      * used to identify the null pixels in the image.
446      * </p>
447      * <p>
448      * The recommended method of recording the null pixels when a lossy
449      * compression algorithm is used is to create an integer data mask with the
450      * same dimensions as the image tile. Set the null pixels to 1 and all the
451      * other pixels to 0, then compress the mask array using a lossless
452      * algorithm such as PLIO or GZIP. Store the compressed byte stream in a
453      * variable-length array column called ’NULL PIXEL MASK’ in the row
454      * corresponding to that image tile. The ZMASKCMP keyword should be used to
455      * record the name of the algorithm used to compress the data mask (e.g.,
456      * RICE 1). The data mask array pixels will be assumed to have the shortest
457      * integer datatype that is supported by the compression algorithm (i.e.,
458      * usually 8-bit bytes).
459      * </p>
460      * <p>
461      * When uncompressing the image tile, the software must check if the
462      * corresponding compressed data mask exists with a length greater than 0,
463      * and if so, then uncompress the mask and set the corresponding undefined
464      * pixels in the image array to the appropriate value (as given by the BLANK
465      * keyword).
466      * </p>
467      */
468     public static final String NULL_PIXEL_MASK_COLUMN = "NULL_PIXEL_MASK_COLUMN";
